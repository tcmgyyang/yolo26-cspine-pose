# Cervical Spine Keypoint Detection with YOLO26-Pose

Code accompanying the paper:

> **<paper title — fill in upon publication>**
> *<authors>*
> *<journal>* (year)
> DOI: <to-be-filled-after-acceptance>

This repository contains the full training, evaluation, ablation, and analysis code for a
cervical-spine keypoint detection model based on a vendored fork of
[Ultralytics](https://github.com/ultralytics/ultralytics). The proposed architecture augments the
YOLO26-pose backbone–neck–head with the **DPCA** channel-attention module and a dynamic upsampling
block, and is trained with an OKS loss whose per-keypoint σ priors are tuned to the anatomical
landmarks of the cervical spine.

The released code reproduces all experiments reported in the paper:

1. **LOSO** (Leave-One-Source-Out) — 6-fold cross-source generalization across the six
   participating clinical sources.
2. **Single-source → all-source transfer matrix** — 6×6 matrix probing how a model trained on a
   single clinical source generalizes to the others.
3. **Module ablations** — exhaustive ablations toggling DPCA placement (positions A/B/C/D and
   their combinations) and the dynamic upsampling block.
4. **Hyper-parameter sweeps** — search over loss weights, augmentation scales, and σ priors.
5. **Per-keypoint error analysis**, **inference-speed benchmarks**, and **ONNX export** for
   deployment.

## Data availability

Patient X-ray images and per-patient keypoint annotations are **not redistributed** in this
repository because of patient-privacy and institutional data-use restrictions. De-identified data
may be made available to researchers under a Data Use Agreement (DUA) with the corresponding
author.

The repository ships:

- The proposed and ablation **architecture YAMLs** under `ultralytics/cfg/models/26/`.
- All **pipeline, ablation, validation, inference, and export scripts** required to regenerate
  splits, train, evaluate, and aggregate results once data is provided.
- Empty `loso_configs/loso/*.yaml` and `loso_configs/single_source/*.yaml` data-config templates.

Image lists referenced by the YAMLs are not redistributed; they will be regenerated by
`step1_make_loso_splits.py` once the dataset is provided locally.

## Repository layout

### 1. Numbered pipeline (main experiments)

| Step | Script | Purpose |
|---|---|---|
| 0 | `step0_prepare_data.py` | Convert raw DICOM/JPG + LabelMe annotations to YOLO pose format. |
| 1 | `step1_make_loso_splits.py` | Generate per-source LOSO and single-source train/val/test image lists. |
| 2 | `step2_run_loso_train.py` | Train baseline + proposed model under 6-fold LOSO. |
| 3 | `step3_run_transfer_train.py` | Train proposed model on each single source; eval on every source. |
| 4 | `step4_collect_results.py` | Aggregate `mAP@50:95` numbers into `LOSO_BASE/PROP` arrays + `TRANSFER` 6×6 matrix. |
| 5 | `step5_demographics.py` | Patient-level demographics report (cohort table). |

### 2. Standalone training scripts

| Script | Purpose |
|---|---|
| `train_cervical_pose.py` | Single-run training of the proposed model on a single dataset YAML. |
| `train_dpca.py` | Train the DPCA-augmented variant in isolation. |
| `train_dpca_ablation.py` | Sweep DPCA module ablations (A/B/C/D and their combinations). |
| `train_ablation_all.py` | Run all module-level ablations (DPCA × dynamic upsampling × MS-OKS σ). |
| `train_generalization.py` | Cross-source generalization training scaffold. |

### 3. Validation and analysis

| Script | Purpose |
|---|---|
| `validate_DPCA.py` | Validate trained DPCA models on a held-out test split. |
| `validate_ablation.py` | Validate every ablation checkpoint and aggregate per-module deltas. |
| `validate_generalization.py` | Validate generalization checkpoints across all sources. |
| `compute_errors.py` | Per-keypoint pixel-error statistics (mean, p95, hit-rate at PCK@τ). |
| `evaluate_parameters{,2,3,4,5,6}.py` | Hyper-parameter sweeps and visualization of σ / lr / loss-weight effects. |
| `ablation_visualization.py` | Render publication-ready ablation comparison figures. |

### 4. Inference and export

| Script | Purpose |
|---|---|
| `infer_yolo.py` | Run inference + render keypoint overlays on input images. |
| `measure_inference_time.py` | End-to-end latency benchmark across input resolutions. |
| `export_onnx.py` | Export trained checkpoints to ONNX for deployment. |

### 5. Auxiliary utilities

| Script | Purpose |
|---|---|

## Custom modules

| Component | Location |
|---|---|
| `DPCA` channel-attention block | `ultralytics/nn/modules/block.py` |
| Dynamic upsampling block | `ultralytics/nn/modules/block.py` |
| Anatomically-tuned σ injection (`make_sigma_callback`) | `step2_run_loso_train.py`, `step3_run_transfer_train.py` |

Architecture YAMLs that wire the modules into different positions of the pose network are under
`ultralytics/cfg/models/26/`:

- `yolo26s-pose.yaml` — baseline.
- `yolo26s-pose-dpca-AC.yaml` — proposed configuration (DPCA at backbone-end + P4 head).
- `yolo26s-pose-dpca-{A,B,C,D,AB,AC-noDS,…}.yaml` — ablations toggling DPCA placement and the
  dynamic upsampling block.

The σ values (`SIGMA_CORNER = 0.05`, `SIGMA_SPINOUS = 0.07`, `SIGMA_LAMINA = 0.10`) correspond to
the vertebra-corner, spinous-process, and laminar-edge keypoint groups respectively, as defined in
the paper. They are injected via Ultralytics' `on_train_start` callback so that the OKS loss
during training reflects the cervical-spine anatomy rather than the COCO-17 default.

## Reproducibility notes

- Training uses `seed=42, deterministic=True, amp=False`. We empirically observed that `seed=0`
  triggered a non-recoverable Pose-head divergence on the LOSO/SX fold; `seed=42` resolves it
  without changing any other hyper-parameter (see comment in `step2_run_loso_train.py`).
- For very small training sets (e.g. one of the smaller sources, 53 images), the default
  `workers=18` can deadlock the DataLoader at epoch boundaries; `step3_run_transfer_train.py` uses
  `workers=4` to avoid this.
- The post-training `model.val()` call in `step2`/`step3` writes to `runs/<task>/<project>/<name>_eval`;
  the path is constructed relative to the script's working directory and can collide with the
  vendored Ultralytics' auto `runs/pose/` prefix. Run pipelines from the repo root, and use
  `_rerun_loso_val.py` / `_rerun_transfer_WJ_val.py` to recover if the inline val step fails.
- Several legacy scripts use absolute paths in their default arguments; they have been rewritten
  to use repo-relative paths in this release. Adjust them to your local layout where necessary.
